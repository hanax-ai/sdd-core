This register records the risks of the governed transition from fixture-backed CENTCOM data to validated, provenance-backed live GitHub data (the first vertical slice: `get_snapshot` live; the six remaining tools explicitly fixture-backed — seven tools total, one migrated first, six remaining, per FR-004).

R-07's identifier is retained for stability of prior citations even though its underlying constitutional question is now resolved (see the R-07 detail).
- **Likelihood / Impact scale**: qualitative **L** (low), **M** (medium), **H** (high). Likelihood is assessed for the risk materializing if the listed mitigations were absent or weakened; impact is assessed for the consequence if it materializes. Where the designed controls change the residual picture materially, the detail entry says so.
- **Owners are roles, not persons**: see the role glossary below. Ownership means accountability for the mitigation being in place and for acting on the trigger signal — it does not confer implementation authority (FR-030).
- **Maintenance**: this register is a planning artifact. Rows are updated through the same governed change process as the sibling artifacts; realized risks become incidents handled per FR-028 and the incident and rollback runbook (task T024), with the register updated afterward.

## Risk Detail

### R-01 — Provenance failure: mislabeled or missing envelope

- **Description**: A result served by any of the seven MCP tools, or a collection consumed by the dashboard, omits the provenance envelope or carries incorrect values in it — most dangerously, `dataMode: live` on fixture-derived content, a missing `governanceWarning` on a non-authoritative result, or `authoritative: true` where authority cannot be established. This is the core dishonesty failure of the transition: it converts demonstration data into apparent operational truth (spec User Story 2).
- **Likelihood**: M — the envelope must be constructed correctly at every serving surface, and the count of surfaces grows with each migrated collection; without a single shared construction path, drift is probable.
- **Impact**: H — a consumer (human or agent) acting on mislabeled governance state is the single most dangerous failure named by the specification.
- **Requirement / control mapping**: FR-008 (envelope minimum fields), FR-009 (non-authority warning), FR-011 (mode-confusion tests); SC-002 (100% envelope coverage), SC-009 (classification from the result alone); contract `contracts/provenance-envelope.md` (v1.2.0, including required constituent `contractVersion`); tasks T004 (envelope contract test), T006 (mode-confusion suite, including the fixture-poisoning canary), T012 (contract schemas in the shared validation seam), T014 (shared envelope construction in the shared server-side data service).
- **Mitigation**: one shared envelope construction and verification path — the shared server-side data service (T014, `src/server/centcom-data/envelope.ts`) consumed by both the dashboard and the MCP endpoint — so labeling cannot fork; contract tests that fail on missing or extra fields, including a missing constituent `contractVersion` (T004); a fixture-poisoning canary — a marker value present only in fixtures that must never appear in any result labeled live (T006); "serving of mislabeled data" defined as a named incident category with a rollback path (FR-028, FR-019).
- **Owner**: Implementing agent (construction and tests); CENTCOM operator (detection in operation).
- **Trigger / monitoring signal**: any mode-confusion or envelope contract test failure; any envelope validation failure at serving time; periodic SC-002/SC-009 sampling below 100%; an incident report of a result whose labeling disagrees with the ingestion record.

### R-02 — Mixed fixture/live results misread as wholly live

- **Description**: During the first slice the system's steady state is mixed: `get_snapshot` is live while five collections (serving six tools) remain fixture-backed. An aggregate view, page-level status, metric, or summary that combines them is presented — or is reasonably perceived — as wholly live, misrepresenting six sevenths of the data (spec User Story 1, acceptance scenario 3; User Story 2, acceptance scenario 2).
- **Likelihood**: M — mixed presentation is the default condition of the entire first-slice period, and aggregate surfaces are precisely where per-collection provenance is easiest to lose.
- **Impact**: H — the misread converts fixture content into apparent live governance state at the surface consumers actually look at.
- **Requirement / control mapping**: FR-010 (mixed must never be presented as wholly live, at any surface), FR-004/FR-007 (six tools verifiably fixture-backed, count stated unambiguously); SC-002, SC-009; tasks T006 (mode-confusion across mixed conditions), T017 (dashboard mixed-state labeling on aggregate views), T019 (fixture envelope enforced on the six tools).
- **Mitigation**: `dataMode: mixed` declared with authority per constituent collection wherever live and fixture content combine; the dashboard's data-mode indicator driven by adapter output rather than hardcoded copy (T016 surfaces mode from data; T017 replaces Phase 1's static "fixture" label); the unambiguous count — seven tools total, one migrated first, six remaining — repeated wherever migration state is described; the test suite fails if any surface presents a mixed state as live (FR-011).
- **Owner**: Implementing agent.
- **Trigger / monitoring signal**: mode-confusion suite failure in the mixed condition; design review of any new or changed aggregate view; SC-009 trials misclassifying a mixed result; a consumer report citing a fixture-backed value as live evidence.

### R-03 — Authentication-versus-authorization gap

- **Description**: The MCP's existing OAuth layer authenticates an app user who is not authorized to read governance data, and authentication is treated as sufficient for access. The spec names this edge case explicitly: authentication without authorization. The gap widens as collections with higher data classification migrate (FR-027).
- **Likelihood**: M — the Phase 1 baseline has authentication only; authorization is a new control that does not exist until implemented, and the failure mode is silent (access succeeds).
- **Impact**: H — unauthorized principals reading live operational governance data; for the first slice the exposure is bounded because the snapshot content is public-class data (spec Q-004, resolved 2026-07-23), but the pattern, once wrong, persists into later slices where classification is higher.
- **Requirement / control mapping**: FR-021 (distinct controls; explicit user-access policy before non-public data — or live operational data beyond the explicitly public-class snapshot fields cleared under Q-004/FR-027 — is exposed), FR-022 (tool-level access control), FR-023 (token audience validation), FR-027 (per-collection data classification before migration); research.md R-5; task T021.
- **Mitigation**: allowlist-first authorization model layered on the existing OAuth surface, backed by the `centcom` access-policy tables (T021); tool-level restriction so individual tools or their live data can be limited to authorized principals; audience validation rejecting tokens issued for other resources; the FR-027 classification decision recorded per collection before it migrates; the serving posture is decided (Agent Zero, 2026-07-23): public dashboard access is limited to explicitly public-class snapshot fields, MCP access stays OAuth-protected, and **no non-public collection migrates until authorization and revocation controls are operational and validated**.
- **Owner**: Security workstream owner.
- **Trigger / monitoring signal**: invocation audit log (FR-026) showing access by principals outside the allowlist; a token with a foreign audience accepted in any test; the readiness-to-request-Gate-2 review finding the access policy undefined or unenforced; any migration proposal for a non-public collection without operational, validated authorization and revocation controls.

### R-04 — Data drift: fixture/live divergence and contract-version drift

- **Description**: Two related drifts. (a) During the mixed period the fixture dataset and live data disagree materially — an expected condition the spec's edge cases anticipate; the danger is unlabeled disagreement, where consumers cross-read fixture and live values as one coherent state. (b) Schema drift between contract versions: the reconciliation producer and one or both consumers disagree about the contract shape or vocabulary, so validated content is misinterpreted at read time.
- **Likelihood**: H — fixture/live divergence is near-certain over any extended mixed period (the fixture is frozen; the repository moves); contract evolution across future slices makes version skew probable without explicit versioning discipline.
- **Impact**: M — labeled divergence is a disclosed, tolerable operating state (the labeling, not the reconciliation of values, is the requirement); the impact rises toward H only when drift is unlabeled or when contract skew silently misinterprets data — which the controls below convert into loud validation failures.
- **Requirement / control mapping**: FR-010 (per-collection labeling makes divergence safe), FR-013 (out-of-vocabulary values fail validation rather than being coerced), FR-015 (snapshot version and commit pinning), FR-003 (validation boundary with recorded failures); `contractVersion` and `datasetVersion` in every envelope, including every mixed-mode constituent (FR-008); SC-003; tasks T012 (strict contract schemas, v1.2.0, `$id` URNs), T011 (schema-compilation validation), T020 (consumer-consistency test).
- **Mitigation**: divergence is handled by labeling, never by reconciliation of values or silent preference; every result — and every mixed-mode constituent — names its contract version, and fixture results their dataset version, so consumers can detect skew from the result alone; contract changes are governed changes to the Contract Version entity, not incidental edits; strict validation converts semantic drift into recorded validation failures instead of served corruption; the schema-compilation step (T011) makes broken or unresolvable cross-references a CI-visible failure rather than a latent skew; the consumer-consistency test proves dashboard and MCP read the same snapshot version identically.
- **Owner**: Contract steward (versioning); Implementing agent (validation enforcement).
- **Trigger / monitoring signal**: T020 divergence between consumers at the same snapshot version; a schema-compilation failure after a contract change; a spike in validation failures following a contract change; a served result whose `contractVersion` differs from the consumer's expected version; a material fixture/live disagreement surfacing in consumer reports without its mixed labeling being noticed.

Accepting Stage C implementation through a weakened gate would remove the control that distinguishes evidence from enforcement. The governing lesson applies verbatim: *local validation is evidence; GitHub CI is enforcement; a PR description is neither.*
- **Likelihood**: L — the gate is operational and its weakening would require a deliberate repository-governance change, which is itself approval-gated and outside this feature's scope.
- **Impact**: H — unchanged from R0: implementation accepted without an enforced CI gate.

### R-06 — Rollback failure: reversion does not restore honest labeling

- **Description**: A directed rollback to fixture-only operation fails its contract — the reversion leaves live-labeled content still being served, serves fixture data still carrying live labels, restores only some of the seven tools, or turns out to require data repair. Rollback is defined as a single governed reversion to fixture-only operation, correctly labeled, with no data repair (FR-019); anything less converts the safety mechanism into a second incident.
- **Likelihood**: M — rollback paths that are not exercised routinely decay; the labeling half of the contract (not merely serving fixtures, but serving them honestly labeled) is the part most easily broken by an incomplete reversion.
- **Impact**: H — rollback is the terminal mitigation for R-01, R-08, R-10, R-16, R-17, and the FR-028 incident categories; if it does not restore honest labeling, the system has no floor.
- **Requirement / control mapping**: FR-019 (single governed reversion, no data repair), FR-018 (labeled fallback hierarchy); SC-005 (rollback verified by the mode-confusion suite passing in fixture-only configuration); tasks T010 (rollback test), T016 (per-collection source selection, which makes fixture-only a configuration state rather than a code change), T024 (incident and rollback runbook).
- **Mitigation**: rollback designed as configuration reversion, not data surgery — the adapter's per-collection source selection (T016) pins all collections back to the fixture dataset (restoring all seven tools to fixture service), which remains available, deterministic, and versioned throughout Phase 2 (spec Assumptions); fixture-only operation reads nothing from PostgreSQL, so rollback holds even when the datastore itself is the incident (see R-16, R-17); the rollback test proves fixture-only configuration returns all seven tools and the dashboard to correctly labeled fixture service; SC-005 makes the mode-confusion suite the verification instrument, so "rolled back" and "honestly labeled" cannot be certified separately; the runbook (T024) makes the reversion a rehearsed governed action with named steps.
- **Owner**: CENTCOM operator (execution); Implementing agent (rollback mechanism and test).
- **Trigger / monitoring signal**: rollback drill or T010 test failure; the mode-confusion suite failing in fixture-only configuration; any post-reversion result carrying `dataMode: live` or lacking the fixture non-authority warning; a reversion procedure found to require manual data edits.

### R-07 — Article II standing — RESOLVED; residual: schema/persistence discipline erosion

- **Status (R1)**: **RESOLVED as a constitutional question; retained monitor-only.** The R0 row recorded an Article II escalation risk under the then-proposed design. That design is retired: per Agent Zero's decision (2026-07-23, responding to independent review Finding 1), **PostgreSQL is the operational system of record** — root Article II's mandated standardized relational store (the existing Supabase project's managed PostgreSQL, in a dedicated non-public `centcom` schema) holding normalized snapshots, ingestion runs, provenance, access-policy state, and audit records; JSON is limited to fixtures, tests, and transient exports (spec Q-002, resolved 2026-07-23; plan.md Constitution Check, root Article II). The former "design avoidance" posture and its escalation trigger no longer exist. The identifier is retained because prior revisions cite the Article II question as "risk R-07".
- **Description (residual)**: The mandated persistence discipline erodes in implementation or operation: a durable JSON/CSV snapshot file or object-store export reappears as a second datastore; `centcom` schema changes are made outside source-controlled migrations; or a separate PostgreSQL instance is introduced without escalation. Each would recreate the Article II violation the decision closed — this time embedded in shipped infrastructure.
- **Likelihood**: L — the decision is recorded at requirement level (FR-015, FR-020) and in every sibling artifact, and no task produces a durable JSON artifact (tasks.md Notes).
- **Impact**: H — a root-constitution violation that must be unwound after the fact, plus the precedent damage of a governed boundary crossed silently.
- **Requirement / control mapping**: spec Q-002 (resolved 2026-07-23); FR-001 (PostgreSQL in the single truth path), FR-015 (snapshots persist in the PostgreSQL system of record), FR-020 (ingestion record in PostgreSQL); task T013 (source-controlled migrations; least-privilege roles); plan.md Constitution Check (root Article II — PASS because PostgreSQL is the mandated store).
- **Mitigation**: JSON is confined to fixtures, tests, and transient exports — no task produces a durable JSON/CSV snapshot file, object-store export, or any second datastore; every `centcom` schema change travels through source-controlled `supabase/migrations/` under review (T013), never console edits; a genuine isolation, capacity, compliance, or deployment blocker justifying a separate PostgreSQL instance is an explicit escalation to Agent Zero (see R-16), never a local choice; acceptance review (T026) verifies no persistence exists beyond the `centcom` PostgreSQL schema and the permitted JSON roles.
- **Owner**: Approving authority (any new-datastore or separate-instance escalation); Implementing agent (detection and escalation duty).
- **Trigger / monitoring signal**: any implementation proposal, dependency addition, or configuration introducing durable storage beyond the `centcom` PostgreSQL schema (durable JSON/CSV files, object storage, platform KV, a second database); a schema change discovered outside `supabase/migrations/`; acceptance review finding persisted state not accounted for by the PostgreSQL schema or the permitted JSON roles.

### R-08 — Stale served as fresh: freshness anchor, threshold, or classification misconfigured

- **Description**: A served result whose freshness anchor is older than the ratified 2-hour staleness threshold is presented without its staleness disclosure — stale data as fresh. Named causes under the ratified freshness model (FR-016; research.md R-4): **threshold misconfiguration** (a value differing from the ratified 2 hours, or differing between environments); **anchor misuse** (freshness computed from anything other than the stored `last_verified_at` — for example the envelope's `observedAt` drawn from the serving clock, or a consumer wall clock substituted for the stored anchor); **classification computed outside the shared data service** (a second, divergent staleness computation); and **clock skew at ingestion** (an observation timestamp regressing against its predecessor).
- **Likelihood**: M — the computation spans a stored anchor, one configured number, and one designated computation point; each is an independent opportunity for silent error.
- **Impact**: H — "a consumer MUST never receive stale data presented as fresh" (FR-016) is a first-order honesty requirement; its violation is indistinguishable, from the consumer's side, from R-01.
- **Requirement / control mapping**: FR-016 (hourly scheduled refresh plus on-demand; stale after 2 hours; disclosure required — ratified by Agent Zero 2026-07-23, closing spec Q-005), FR-018 (last-known-good served *with* staleness disclosure); SC-004; research.md R-4; data-model.md state model S-02; tasks T008 (failure-path tests), T014 (shared data service computes the classification).
- **Mitigation**: a single stored anchor — `last_verified_at`, the `observed_at` of the latest successful ingestion run (`published` or `no-change`), persisted in PostgreSQL — so a no-change run advances freshness via the run record without fabricating a content change; staleness classification is computed at serving time **only** by the shared data service, comparing the stored anchor against the serving clock — the sole permitted serving-time clock use in the design; the exhaustive list of serving-time-varying response fields is `stale` and (when stale) `thresholdExceededBy` — every other field, including the envelope's `observedAt`, derives from stored state; timestamp monotonicity is enforced at ingestion validation, backed by single-flight run serialization (`data-model.md` E-02 — concurrent scheduled and on-demand runs cannot interleave the observe → publish sequence) and the database-side monotonic guard (E-08, T013 migration — the store rejects an anchor regression outright), so clock-skew regression and out-of-order concurrent runs are rejected in depth, not silently accepted; the threshold is the single ratified number (2 hours), not a per-surface constant; failure-path tests assert the disclosure appears exactly when the threshold is crossed.
- **Owner**: CENTCOM operator (threshold configuration and monitoring); Implementing agent (computation and tests).
- **Trigger / monitoring signal**: monitoring (FR-028) showing anchor regression or a staleness signal disagreeing with the ingestion record; threshold configuration differing between environments or from the ratified 2-hour value; a result older than the threshold observed without its disclosure in SC-004 verification; any serving-path code other than the shared data service found reading a clock.

### R-09 — Source rewrite: force-push or unresolvable head of the governed repository

- **Description**: The governed repository's default branch is force-pushed or rewritten, or its head SHA becomes unresolvable between reconciliation runs (a named spec edge case). Recorded source commit SHAs in prior provenance envelopes may then no longer resolve to reachable commits, breaking the auditor's walk from served value to source revision (User Story 4; SC-001), and the next run observes a non-fast-forward head change.
- **Likelihood**: L — the governed repository is `hanax-ai/sdd-core`, whose main branch moves through governed merges; history rewrite of governed main would itself be a governance event, not routine operation.
- **Impact**: M — service continuity is unaffected (the next run simply observes the new head), but retroactive verifiability of previously served results is damaged, and the event signals a governance breach upstream of this capability.
- **Requirement / control mapping**: FR-005 (snapshot sourced from the governed repository's observable state), FR-015 (commit pinning and snapshot versioning), FR-017 (defined behavior for contradictory source records), FR-020 (append-oriented ingestion record in PostgreSQL); SC-001 (retroactive SHA verification); spec Edge Cases.
- **Mitigation**: the ingestion record permanently preserves what was observed and when, independent of the source's later history — a rewritten source cannot rewrite the observation trail; an unresolvable head at reconciliation time is handled as a retrieval/validation failure per FR-018 (serve the last published snapshot with staleness disclosure), never by guessing; historical envelopes are not repaired — an unresolvable recorded SHA is disclosed as a historical observation whose source has since been rewritten; the event is escalated to the approving authority as a governance incident concerning the source repository, not merely a reconciliation anomaly.
- **Owner**: CENTCOM operator (detection); Approving authority (governance response).
- **Trigger / monitoring signal**: a reconciliation run observing a head change that is not a descendant of the previously recorded head; SHA resolution failure during reconciliation or during SC-001 audit sampling; an auditor's provenance walk failing to resolve a recorded commit in the named repository.

### R-10 — Credential leak in reconciliation tooling

- **Description**: The live-source credential used by the server-side reconciliation component (a read-only token toward the governed repository, held in the Edge Function's server-side environment) leaks — into client-delivered code, browser storage, URLs, logs, error messages, committed configuration, MCP results, or the ingestion record itself. The reconciliation tooling is the one place in the design where a secret necessarily exists, making it the concentrated leak surface.
- **Likelihood**: M — credential leakage through logs, error paths, and committed configuration is among the most common real-world security failures, and reconciliation tooling exercises all three.
- **Impact**: H — even a read-only, minimally scoped token grants an attacker authenticated read posture and burns trust in the pipeline; the incident response (revocation, rotation, audit of exposure window) is costly, and a broader-scoped token by misconfiguration would be worse.
- **Requirement / control mapping**: FR-024 (server-side only, read-only, minimally scoped; never in any client-visible or committed location), FR-028 (credential leak as a named incident category with a rollback path); SC-007 (zero secret material at acceptance, by inspection and automated scan); tasks T015 (reconciliation Edge Function with server-side-only token), T022 (automated secret-scan verification, including Edge Function environment/secret handling).
- **Mitigation**: credentials held server-side only — in the Edge Function's environment, never in repository files, migrations, or client-reachable code — and scoped read-only and minimally, so the blast radius of a leak is bounded by construction; an automated scan over client bundles, results, logs, and committed configuration is acceptance evidence (T022, SC-007), not a one-time courtesy; error messages and the ingestion record are specified to carry failure context without token contents (FR-020, FR-026); the incident procedure names credential leak explicitly, with revocation and the FR-019 rollback path (fixture-only operation requires no live credential at all — rollback removes the secret from the serving equation entirely).
- **Owner**: Security workstream owner (controls); CENTCOM operator (incident response).
- **Trigger / monitoring signal**: any secret-scan finding; a token observed in a client bundle, log line, error message, MCP result, migration, or commit; a source-platform notification of an exposed token; audit-log anomalies consistent with credential misuse.

### R-11 — Rate-limit exhaustion degrading reconciliation

- **Description**: The live source rate-limits or throttles reconciliation (a named spec edge case), or MCP invocation volume exhausts serving-side limits. Repeated throttled runs push the freshness anchor toward and past the 2-hour staleness threshold; a badly handled exhaustion could crash a run or silently skip an observation.
- **Likelihood**: M — rate limiting is a routine behavior of the live source, guaranteed to occur eventually; the hourly cadence makes *sustained* exhaustion unlikely but not impossible (shared token budgets, retry storms).
- **Impact**: M — the designed consequence is bounded: reconciliation degrades to serving the last published snapshot with staleness disclosure (FR-018), so consumers lose freshness, never correctness or honesty. The impact rises only if degradation is mishandled — which is precisely what the controls forbid.
- **Requirement / control mapping**: FR-025 (MCP invocation limits; reconciliation respects source limits with defined backoff; exhaustion degrades per FR-018, never crash or silent skip), FR-016 (staleness disclosure as freshness decays), FR-018; SC-004; task T023; research.md R-4 (the hourly cadence sits far inside the source's documented rate budget).
- **Mitigation**: reconciliation cadence sized well below the live source's documented limits, with defined backoff on throttling responses; exhaustion is a first-class run outcome recorded in the ingestion record (FR-020), not an exception path; serving degrades along the FR-018 hierarchy — last-known-good with staleness disclosure — so the consumer-visible effect of throttling is honest staleness, not error; MCP invocation rate limits protect the serving side independently of reconciliation.
- **Owner**: CENTCOM operator.
- **Trigger / monitoring signal**: an ingestion-record failure streak with a rate-limit error class (FR-028 monitoring); staleness approaching the threshold because successive runs were throttled; MCP invocation volume anomalies against the configured limits.

### R-12 — MCP serving divergence from the dashboard (competing implementation)

- **Description**: The MCP serving path acquires an independent fetch, normalization, or computation step — a second implementation of the truth path — so dashboard and MCP outputs diverge for the same collection. The spec is explicit that this is a specification violation, not an implementation detail (FR-002; Quick Guidelines). Under R1 the MCP source is bound to exact dashboard-repo paths — the MCP endpoint route `src/routes/mcp.ts` and tool modules `src/server/mcp/`, reading the shared server-side data service `src/server/centcom-data/` (plan.md MCP source binding; tasks T014/T018) — which closes the R0 window of an unbound serving locus; the residual elevation is that the binding rests on a recorded deployment assumption whose failure would relocate the MCP source (see R-15).
- **Likelihood**: M — two consumers with separate delivery pressures naturally drift toward local convenience implementations.
- **Impact**: H — divergent outputs destroy the one-truth-path guarantee (FR-001) and, worse, do so quietly: each consumer looks internally consistent while the pair disagrees.
- **Requirement / control mapping**: FR-001 (single normalized path — GitHub sources → validation/normalization → PostgreSQL system of record → shared data model → both consumers), FR-002 (no competing implementations or separate sources of truth); SC-003 (zero divergence at the same snapshot version); User Story 4 acceptance scenario 2; tasks T014 (the shared data service — the single normalization/serving layer both consumers read), T018 (`get_snapshot` served from the shared data service, no independent fetch), T020 (consumer-consistency test).
- **Mitigation**: the shared server-side data service is the only sanctioned read surface — neither consumer calls the live source or PostgreSQL independently (the dashboard hydrates the Phase 1 adapter from the service's transient versioned response; the MCP endpoint reads the same service); the consumer-consistency test compares dashboard and MCP outputs for the same collection at the same snapshot version and fails on any divergence; T018 is held to the contract files regardless of where the binding decision of R-15 lands — either home consumes the same service layer; any fetch call or normalization logic appearing in the MCP path is a review-blocking finding, not a style issue.
- **Owner**: Implementing agent (both serving paths land in the dashboard-scoped implementation).
- **Trigger / monitoring signal**: T020 divergence; an `[EXT]` change introducing network retrieval or transformation logic in the MCP serving path; review finding a second copy of any normalization rule; envelope `sourceCommitSha` or snapshot version disagreeing between consumers for the same request window.

### R-13 — Unbounded mixed period eroding trust

- **Description**: Spec Q-003 resolves that the mixed fixture/live period has no fixed end date — it closes collection-by-collection through future governed slices, each requiring new authorization. The risk of that deliberate unboundedness: prolonged mixed operation normalizes the warnings. Consumers habituate to `authoritative: false` and the non-authority warning ("warning fatigue"), begin treating fixture-backed collections as adequate operational truth, or lose confidence in the surface as a whole because most of it remains labeled non-authoritative indefinitely.
- **Likelihood**: M — habituation to persistent warnings is a well-established failure pattern, and nothing in the first slice's scope forces a second migration on any schedule.
- **Impact**: M — no data is mislabeled (the labeling discipline holds); the erosion is in consumer behavior and confidence, which degrades the capability's value and raises the chance that a fixture-derived value informs a real decision despite its warning.
- **Requirement / control mapping**: FR-009 (explicit warnings on non-authoritative results), FR-010 (mixed labeling as the control that makes an unbounded mixed period safe — spec Q-003's own reasoning); SC-002, SC-009; spec Assumptions (the mixed period is an intended, disclosed operating state).
- **Mitigation**: the labeling discipline is the primary control and is continuously verified (SC-002/SC-009 are not one-time acceptance checks — the mode-confusion suite runs for the life of the capability); each subsequent migration requires a new governed authorization, which creates natural review points at which the approving authority weighs the mixed period's duration; the Coordinator tracks the migration backlog so "no fixed end date" never becomes "no one is watching"; SC-009 classification trials repeated periodically detect whether results still communicate their mode effectively to fresh consumers.
- **Owner**: Approving authority (migration sequencing decisions); Coordinator (backlog tracking and periodic review).
- **Trigger / monitoring signal**: a consumer observed acting on fixture-backed data despite its warning; an extended interval with no migration decision on record; periodic SC-009 trials showing classification accuracy declining; feedback that surfaces are being ignored because "most of it is fixture anyway".

### R-14 — Implementation-authority inference from planning artifacts

- **Description**: A future session, agent, or reviewer infers implementation authority from the existence, completeness, or approval of the planning artifacts in this feature folder — treating an approved plan, a green baseline, a merged planning PR, a working demo, or a fully satisfied readiness checklist as permission to begin `[EXT]` implementation. User Story 5 names the three-stage boundary the capability's own definition must encode: **Stage A** — planning approval; **Stage B** — Agent Zero's explicit Gate 2 implementation directive ("Approved for implementation: \<exact spec/plan\>"), which **IS** the implementation authorization; **Stage C** — implementation, then validation and acceptance review. No agent, green test, merged PR, or working demo substitutes for those approvals, and no implementation evidence is ever a prerequisite to the Gate 2 directive itself.
- **Likelihood**: M — this inference is the natural failure mode of multi-session, multi-agent work: each artifact individually invites "the plan is approved, so proceed"; the risk recurs at every session boundary.
- **Impact**: H — unauthorized implementation violates WO-P2-AUTHOR-001's authorization scope (§1, §9) and WO-P2-AUTHOR-001-R1's stop conditions, bypasses Gate 2 (Stage B), and lands ungoverned changes in a governed repository.
- **Requirement / control mapping**: FR-030 (planning artifacts must not be construed as implementation authority; the three-stage model); User Story 5 and its acceptance scenarios; SC-008 (Stage C acceptance gating); task T001 (verbatim capture of the Gate 2 directive blocks every `[EXT]` task); `implementation-authorization-boundary.md` (the dedicated boundary artifact, this register's primary mitigation reference); `gate-2-entry-criteria.md` (readiness-to-request-Gate-2 criteria — pre-implementation items only); the authoring-only banner in tasks.md.
- **Mitigation**: mitigated in depth by `implementation-authorization-boundary.md`, which exists precisely to make this inference impossible to make honestly; every sibling artifact carries the boundary statement in-band (spec status line, plan header, tasks banner), so no single-document reading misses it; T001 makes the Gate 2 directive a recorded artifact (`records/implementation-authorization.md`) with the directive captured verbatim — its absence is machine-checkable and blocks all `[EXT]` execution; the readiness criteria contain only pre-implementation items and state explicitly that satisfying all of them does not constitute Gate 2 — they define readiness to *request* the directive, nothing more; the R1 directive's stop conditions end this remediation work order at presentation for independent re-review.
- **Owner**: Approving authority (the only role that can grant implementation authority — Gate 2 is its explicit directive); Coordinator (work-order gatekeeping — verifying the T001 record exists before routing implementation work).
- **Trigger / monitoring signal**: any `[EXT]` task started, or any dashboard-repo change proposed, without `records/implementation-authorization.md` populated; a pull request or work product citing this feature folder as its authorization; a handoff or session summary asserting implementation may proceed without citing the recorded Gate 2 directive.

### R-15 — MCP deployment binding: platform cannot deploy the MCP from repository-visible source *(added under R1)*

- **Description**: The MCP's source-controlled home is bound (plan.md MCP source binding) to the dashboard repository: the MCP endpoint route `[EXT] src/routes/mcp.ts` (TanStack server-handler route implementing the MCP Streamable HTTP endpoint) with tool modules under `[EXT] src/server/mcp/`, deployed through the dashboard's existing Lovable lineage. The binding rests on a **recorded assumption**: that the platform deploys repo-visible server routes to the hosted endpoint — evidenced by the already-shipped server-handler route `src/routes/sitemap[.]xml.ts` — AND that the hosted `/mcp` path can be routed to repository code. The risk: the platform cannot deploy the MCP endpoint from repository-visible source, leaving the hosted MCP surface without a source-controlled, reviewable home and the binding unexecutable as designed.
- **Likelihood**: M — the sitemap-route evidence supports server-route deployment generally, but the `/mcp` path routing specifically has not yet been validated.
- **Impact**: H — without a repository-visible source binding, the one-truth-path guarantee (FR-001/FR-002) cannot be enforced by review or test on the MCP side, and tasks T018/T019/T021/T023 have no executable home; an externally hosted MCP with unresolved canonical source is exactly the condition the independent review found blocking.
- **Requirement / control mapping**: plan.md MCP source binding (bound paths + recorded assumption); `gate-2-entry-criteria.md` (the binding-validation readiness-to-request-Gate-2 item); tasks.md Notes (binding and escalation recorded in-band); tasks T018 (MCP endpoint route).
- **Mitigation**: validating the deployment assumption is an explicit readiness-to-request-Gate-2 item — it completes, or fails and escalates, **before** planning approval concludes, never after implementation begins; the escalation path is pre-approved in the planning record: if the platform cannot deploy the MCP from repository-visible source, a **separate service repository** is proposed and escalated to Agent Zero BEFORE planning approval; either outcome preserves the shared data service (T014) as the single normalization layer, so the one-truth-path architecture survives the binding decision unchanged; the binding names exact paths, so the validation is a concrete check, not a judgment call.
- **Owner**: Coordinator (validation tracking and escalation routing); Approving authority (the separate-service-repository escalation decision).
- **Trigger / monitoring signal**: a validation attempt showing the hosted `/mcp` path cannot be served from `src/routes/mcp.ts`; platform documentation or observed behavior contradicting the server-route deployment evidence; the readiness review reaching the MCP-binding item without recorded validation evidence or a recorded escalation.

### R-16 — Supabase single-project coupling: OAuth, data, and cron share one blast radius *(added under R1)*

- **Description**: The design deliberately reuses the existing Supabase project — the one already providing MCP OAuth — as the PostgreSQL system of record (the `centcom` schema) and as the reconciliation host (Edge Function invoked by Supabase Cron). Three concerns therefore share one project: authentication, operational data, and scheduling. The blast radius risk: a failure, misconfiguration, quota exhaustion, or compromise in one concern degrades the others — a project-level outage takes down authentication and data serving together; an over-privileged role or leaked project credential bridges OAuth state and operational data; reconciliation load competes with serving for project limits.
- **Likelihood**: M — shared-platform coupling failures are routine operational events, and the coupling is a deliberate design economy rather than an accident.
- **Impact**: M — degradation is bounded by the FR-018 fallback ladder: fixture-only service reads nothing from Supabase, so even total project loss degrades to correctly labeled fixture service, never dishonest serving. The impact rises toward H only if privilege separation fails (data exposure across concerns) — which the controls below are designed to prevent.
- **Requirement / control mapping**: spec Assumptions (the existing Supabase project's managed PostgreSQL as system of record); FR-018 (labeled fallback hierarchy), FR-019 (fixture-only rollback), FR-021–FR-024 (authorization and credential controls); task T013 (dedicated `centcom` schema; least-privilege roles).
- **Mitigation**: CENTCOM operational data lives in a dedicated, non-public `centcom` application schema — it never mingles with OAuth or platform tables; least-privilege roles separate the write path (reconciliation writer role) from the read path (serving read-only role), so no single credential spans concerns and the compromise of one path does not open the other; a concrete isolation, capacity, compliance, or deployment blocker is a recorded **escalation to Agent Zero proposing a separate PostgreSQL instance/project** — never a silent local migration (and never a silent continuation once a blocker is real); rollback to fixture-only operation (FR-019) removes the Supabase dependency from serving entirely, providing a floor independent of the project's health.
- **Owner**: Security workstream owner (schema and privilege separation); Approving authority (separate-instance escalation decision); CENTCOM operator (operational detection).
- **Trigger / monitoring signal**: a role or credential observed with privileges spanning the `centcom` schema and non-CENTCOM project resources; project-level quota or limit pressure attributable to reconciliation or serving; an incident in one concern observably degrading another; any isolation, capacity, compliance, or deployment blocker surfacing during implementation or operation.

### R-17 — PostgreSQL schema migration failure: defective or irreversible `centcom` schema change *(added under R1)*

- **Description**: The `centcom` schema evolves through source-controlled migrations (T013). A defective migration — wrong DDL, a data-destructive change, a half-applied failure, or a change with no workable reversal — corrupts or strands operational state (snapshot versions, ingestion runs, access-policy state, audit records) or leaves environments schema-divergent. Unlike the FR-019 serving rollback (a configuration reversion), reversing a schema change is a data-layer operation with its own failure modes; conflating the two would overstate the safety floor.
- **Likelihood**: M — schema evolution is certain across slices, and migration defects are a routine class of operational failure.
- **Impact**: H — the ingestion record is the audit trail grounding retroactive verification (SC-001, User Story 4); its corruption breaks verifiability of previously served results, and a stranded schema blocks both the write path (reconciliation) and the PostgreSQL read paths.
- **Requirement / control mapping**: task T013 (source-controlled migrations; append-oriented structures); `gate-2-entry-criteria.md` (PostgreSQL schema design and migration plan reviewed as a readiness-to-request-Gate-2 item); FR-020 (append-oriented ingestion record), FR-018 (serving continuity when the store is unhealthy), FR-019 (fixture-only rollback requires no data repair); task T008 (the PostgreSQL-unavailable failure path is a tested serving condition).
- **Mitigation**: every schema change is a reviewed, source-controlled migration in `supabase/migrations/` — never a console edit; the schema design and migration plan are reviewed **before Gate 2 is requested**, so the first schema lands under scrutiny rather than delivery pressure; append-oriented structures (runs, snapshot versions, audit records) make destructive rewrites an anomaly reviews and monitoring can catch, not a normal operation; each migration states its reversal or restore procedure before application; serving during a migration failure follows FR-018 — last published snapshot with staleness disclosure, then labeled fixture fallback — so a stranded schema never produces a blank surface or a mislabeled result (T008); the FR-019 fixture-only rollback is independent of PostgreSQL health, so the honest-labeling floor survives even an unrecoverable schema incident.
- **Owner**: Implementing agent (migration authoring and review); CENTCOM operator (application and recovery).
- **Trigger / monitoring signal**: a migration failing or half-applying in any environment; environments observed at divergent schema versions; a proposed migration lacking a stated reversal or restore procedure; monitoring or SC-001 audit sampling showing ingestion-run or snapshot rows lost or rewritten.

### R-18 — Reconciliation scheduler reliability: missed runs drive staleness *(added under R1)*

- **Description**: Reconciliation is a separate server-side runtime component — a source-controlled Supabase Edge Function invoked hourly by Supabase Cron, plus Agent Zero's on-demand refresh, both invoking the same ingestion operation (T015). If the cron schedule silently stops firing, the Edge Function fails repeatedly or times out, or the cron registration is lost (for example dropped by a later migration), no new runs advance the freshness anchor and every served result decays toward and past the 2-hour staleness threshold. The failure mode is silent by default: nothing runs, and serving continues without error.
- **Likelihood**: M — scheduler and function failures are routine operational events, and a stopped schedule produces no serving-side error to notice.
- **Impact**: M — the designed consequence is honest staleness, not dishonesty: serving continues from the last published snapshot and discloses `stale: true` once the threshold passes (FR-016/FR-018), so consumers lose freshness, never correctness. The impact rises only if the staleness disclosure itself fails — which is R-08's territory, covered by its controls.
- **Requirement / control mapping**: FR-016 (hourly cadence + on-demand; 2-hour threshold), FR-018 (labeled degradation), FR-020 (run outcomes recorded, including failures), FR-028 (ingestion health monitoring); tasks T015 (Edge Function + cron registration migration), T025 (monitoring: last successful run / `last_verified_at`, failure streak, staleness).
- **Mitigation**: monitoring reads ingestion health directly from the run and audit tables — last successful run, failure streak, anchor age (T025, FR-028) — so a silent scheduler becomes loudly visible; the on-demand refresh invokes the **same** ingestion operation as the schedule, providing an immediate recovery path that needs no redeployment; failed and throttled runs are first-class recorded outcomes (FR-020), never silent skips; the cron registration is itself a source-controlled migration, so its loss is a reviewable change rather than invisible configuration drift; while the schedule is being restored, R-08's controls keep consumers honestly informed of the decaying freshness.
- **Owner**: CENTCOM operator (detection and recovery); Implementing agent (monitoring and cron-registration mechanisms).
- **Trigger / monitoring signal**: anchor age approaching the 2-hour threshold with no recorded run in the interval; a failure streak in the ingestion record; the cron registration absent after a migration; on-demand refresh succeeding while scheduled runs do not occur (a scheduler-specific fault signature).
